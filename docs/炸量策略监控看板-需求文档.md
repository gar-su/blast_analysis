# 炸量策略监控看板 — 需求文档
## 1. 项目背景
当前有一套炸量广告回测分析脚本（`blast_analyzer.py` + `blast_backtest_report.py`），能基于历史数据，按炸前指标（消耗、ROI、用户数等）筛选广告子集，观察其炸后表现。
但现有分析流程存在三个问题：
1. **不可见** — 分析结论藏在 CSV 和 Python 脚本里，管理层和运营看不到
2. **不可持续** — 每次看数据都要手动跑脚本，缺乏持续监控
3. **不可对比** — 想按不同维度拆解炸后表现，要反复改代码重跑
需要一个看板——**监控炸量整体表现，按维度拆解查看**——让决策者和执行者能直观了解炸量效果。
## 2. 项目目标
构建一个「炸量广告筛选回测看板」，实现：
- **表现监控**：持续追踪整体炸量效果 —— 盈亏、ROI 趋势、零回收率
- **趋势观察**：按消耗段、ROI 段、零回收分布等维度拆解炸后表现
- **一键导出**：核心数据可截图/导出用于周报汇报
## 3. 用户角色
| 角色 | 使用频率 | 关注点 |
|------|----------|--------|
| 投放运营人员 | 每天 | 整体炸量表现、零回收趋势、各维度分层表现 |
| 产品/业务负责人 | 周/月 | 炸量策略整体盈亏、有哪些盈利窗口、ROI 走势 |
## 4. 看板定位
**纯事后分析，不联动实时炸量决策。** 看板的核心交互模式是「打开即看」：进入页面后直接展示全量炸后表现的概览，通过卡片和图表按维度拆解。
## 5. 概览页卡片设计
概览页通过 **三张故事卡片** 回答三个递进问题：规模 → 效率变化 → 空耗。每张卡片包含一个 Hero 数字（含公式/对比说明）+ 若干辅助指标。
### 卡片 1：命中规模
- **Hero**：炸后毛利润，副行展示公式（总回收 − 0.4 × 总消耗）及命中广告总数
- **辅助**：炸后总回收 / 炸后总用户 / 炸后付费用户
- **回答**：「炸量命中了多少广告，整体赚不赚钱」
### 卡片 2：效率变化
- **Hero**：ROI 变化（炸前 → 炸后），副行展示 pp 差值
- **辅助**：CPI 变化（炸前 → 炸后）/ 付费率变化（炸前 → 炸后）
- **回答**：「炸量对广告效率的冲击有多大」
### 卡片 3：空耗情况
- **Hero**：炸后纯空耗广告占比，副行展示分子/分母（空耗条数 / 总命中条数）
- **辅助**：纯空耗广告炸后消耗 / 炸后亏损广告数及占比 / 炸后盈利广告数及占比
- **回答**：「多少广告炸后完全白烧，浪费了多少消耗」
## 6. 功能需求
### 6.1 场景一：整体表现监控
**目标**：打开看板，5 秒内判断「炸量整体赚不赚钱」。
**功能点**：
- **概览卡片行**：三张故事卡片（命中规模、效率变化、空耗情况），每张含一个 Hero 数字（含公式/对比说明）+ 若干辅助指标
- **趋势图表**：四张概览图
  - ROI 趋势（折线图，炸前 vs 炸后，按日期）
  - 各消耗段 ROI 变化（柱状图，按炸前消耗分段）
  - 各炸前 ROI 段 ROI 变化（柱状图，按炸前 ROI 分段）
  - 零回收按消耗段分布（堆叠柱状图）
- **图表标注**：每张图下方标注横纵轴含义及图例说明（如何读图）
- **默认时间窗口**：最近 7 天，可切换为最近 14 天/最近 30 天
- **盈亏线显示**：固定展示 40%
### 6.2 场景二：周/月汇报
**目标**：截图一张就能放进周报。
**功能点**：
- **汇总卡片行**：核心指标以大数字卡片排列在顶部，一屏截完
- **周期切换**：本周/本月/自定义时间段
- **环比变化**：每个指标旁边显示「较上周期 +X%/-X%」
- **导出按钮**：一键导出当前视图为图片或 PDF
### 6.3 全局交互：下钻路径
```
全局总览（盈亏 + ROI 趋势 + 分层分布）
  └→ 点击某分层柱 → 该层广告列表
      └→ 点击某条广告 → 单广告详情
          ├ 炸前/炸后指标对比
          ├ 小时级回收曲线
          └ 基础信号（充值次数/集中度/尾段异常）
```

## 8. 数据需求
### 8.1 必需数据字段
| 字段 | 说明 |
|------|------|
| campaign_id | 广告 ID |
| campaign_name / link_id | 广告名称 / 投放链接 |
| blast_time | 炸量触发时刻 |
| pre_cost / post_cost | 炸前/炸后消耗 |
| pre_recharge / post_recharge | 炸前/炸后回收金额 |
| pre_users / post_users | 炸前/炸后用户数 |
| pre_pay / post_pay | 炸前/炸后付费用户数 |
| rule_id | 触发来源规则 ID |
| blast_hour_seq | 炸量触发的第几小时 |
### 8.2 数据来源
研发侧可直接查询数据仓库。参考现有 API：
- **ROI 小时数据**：`admin.netshort.com/prod-api/put/dashboard/hours/roi`
- **Rule Log**：`admin.netshort.com/prod-api/rule/controlRuleLog/queryRunLogByPage`
### 8.3 更新频率
| 数据 | 更新频率 |
|------|----------|
| 概览指标卡片 | 近实时（1 小时延迟可接受） |
| 趋势图 | 每天刷新 |
### 8.4 底层数据表
需要研发建好宽表或视图，至少包含：
- `campaign_id`, `blast_time`
- `pre_cost`, `pre_recharge`, `pre_users`, `pre_pay_users`
- `post_cost`, `post_recharge`, `post_users`, `post_pay_users`
- 预计算指标：`pre_roi`, `post_roi`, `pre_cpi`, `post_cpi`, `pre_pay_rate`, `post_pay_rate`
- `profit` = `post_recharge − 盈亏线比例 × post_cost`
**盈亏线比例为可配置参数**（默认 0.4，即 ROI = 40%）。
## 9. 分析逻辑参考
### 9.1 盈亏计算
```
盈利 = rechargeAmount − 盈亏线比例 × cost
```
盈亏线比例为可配置参数。
### 9.2 分层维度
- **消耗段**：$0-30, $30-60, $60-100, $100-200, $200-300, $300-500, $500+
- **炸前 ROI 段**：0-30%, 30-60%, 60-100%, 100-200%, 200%+
- **用户规模段**：0-5, 5-20, 20-50, 50-100, 100+
### 9.3 零回收定义
炸后 `rechargeAmount == 0` 的广告。看板需展示其消耗浪费金额（即这些广告的炸后消耗之和）。
## 10. 技术约束
- 研发侧自行选择技术栈，不指定框架
- 需支持最近 90 天的数据回溯
- 页面加载时间 < 3 秒
- 图表库推荐使用支持交互下钻的（ECharts / Plotly / AntV）
## 11. 验收标准
1. 管理层打开看板，5 秒内能说出「炸量策略现在赚不赚钱」
2. 每张图表下方有清晰的读图说明（横纵轴含义、图例）
3. 三张故事卡片覆盖规模、效率变化、空耗三个维度
4. 核心数字与 `blast_backtest_report.py` 的输出一致（同数据源前提下）
